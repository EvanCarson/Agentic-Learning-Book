  - prompt: "What makes a parser 'total' in the context of structured output?"
    options:
      - "It only produces a defined action when the model returns perfectly formatted output"
      - "It handles every possible response, including malformed ones, mapping each to a known safe action"
      - "It rejects any response that does not exactly match the expected format and raises an exception"
      - "It strips all whitespace before returning the raw model string"
    answerIndex: 1
    explanation: "A total parser covers every case, including malformed output, and always returns a defined action rather than crashing. Option (a) is exactly the failure a total parser eliminates: a parser that only works on perfectly formatted input is a partial parser - it must map the current malformed response to a safe action without depending on perfect formatting (and is orthogonal to whether you also retry). Raising an exception or returning raw text violates the same safety contract."

    answerIndex: 1
    explanation: "A robust parser that normalizes variants (case, spacing) absorbs drift; regression tests catch regressions immediately. Longer prompts add cost and new failure modes without actually pinning behavior. Option (a) is wrong because adjectives do not pin output format reliably - the lesson explicitly states the fix is a declared grammar and test suite, not ever-more-forceful wording. Option (c) is wrong because pinning to an old version forever is brittle and not what the lesson advises - the durable fix is a parser and regression tests that absorb drift across versions, so you can upgrade safely."
